# Next.js + Bun + PM2 GitLab CI/CD 구축 회고

### --- 단순 자동화를 넘어, 배포 아키텍처를 설계

최근 Next.js 프로젝트에 GitLab CI/CD를 구축하면서\
단순히 "자동 배포를 만들었다" 수준이 아니라,

-   빌드 환경과 런타임 환경의 분리
-   의존성 전략 설계
-   타입 체크와 빌드 정책
-   배포 실패 안전성 보장
-   트래픽 운영 관점에서의 구조 안정성

까지 깊이 고민하게 되었다.

이번 글은 단순 문제 해결 기록이 아니라,\
**CI/CD 아키텍처를 어떻게 설계해야 하는지에 대한 기술적 회고**다.

------------------------------------------------------------------------

# 1. 초기 구조 --- 서버에서 빌드하는 방식

처음 구성은 단순했다.

GitLab Runner\
↓ (소스 업로드)\
Server\
├─ bun install\
├─ bun run build\
└─ pm2 start

CI는 소스를 업로드만 하고,\
실제 빌드는 서버에서 수행했다.

### 장점

-   설정이 단순함
-   CI 설정이 간결함

### 하지만 곧 드러난 문제

-   서버 리소스를 빌드가 사용
-   devDependencies 전략 충돌
-   빌드 실패해도 프로세스가 실행되는 문제
-   빌드 환경이 서버 상태에 종속됨

이 구조는 "동작은 하지만 안정적이지 않은 구조"였다.

------------------------------------------------------------------------

# 2. devDependencies와 빌드 실패 문제

서버에서 다음과 같이 실행했다.

```
bun install --production
```

문제는 Next.js 14의 빌드 방식이었다.

Next.js는 단순 번들링만 하지 않는다.

-   TypeScript 타입 검사
-   ESLint 검사
-   환경 변수 검증

즉, devDependencies가 빌드에 필요하다.

실제 발생한 에러:

Could not find a declaration file for module 'file-saver'

원인:

-   @types/file-saver는 devDependencies
-   --production 옵션 때문에 설치되지 않음
-   Next build는 타입 체크 수행
-   빌드 실패

### 핵심 인사이트

> 빌드 환경은 런타임 환경과 완전히 다르다.

------------------------------------------------------------------------

# 3. 실패했는데도 배포 성공? --- set -e의 중요성

초기 스크립트에는 set -e가 없었다.

```
bun run build
pm2 start ...
```

빌드가 실패해도 쉘은 다음 명령을 계속 실행한다.

결과:

-   빌드 실패
-   pm2는 실행됨
-   CI는 성공으로 표시됨

해결:

```
set -euo pipefail
```

### 교훈

> CI/CD는 성공 자동화가 아니라, 실패를 정확히 감지하는 시스템이다.

------------------------------------------------------------------------

# 4. Husky 문제 --- 의존성은 설정까지 포함한다

git commit 시 발생한 경고:

husky - DEPRECATED

원인:

-   husky v9 사용
-   pre-commit 파일은 구버전 형식

의존성 업데이트는 단순 버전 변경이 아니다.\
설정 방식, 동작 구조까지 함께 바뀐다.

------------------------------------------------------------------------

# 5. 구조 개선 --- CI에서 빌드하도록 변경

개선 후 구조:

GitLab CI\
├─ bun install\
├─ bun run build\
└─ build 결과물 artifact 저장

Deploy 단계\
└─ build 산출물만 서버에 업로드

Server\
├─ bun install --production\
└─ pm2 start

### 변화의 핵심

| 항목 | 이전  |  이후 |
| ---- |---- |----- |
|빌드 위치  |         서버 |  CI |
타입 검증    |      서버 |  CI |
서버 리소스 사용 |  높음  | 낮음 |
배포 안정성   |     낮음  | 높음 |

------------------------------------------------------------------------

# 6. 기술적으로 깊게 보는 아키텍처 분석

## Build Artifact 중심 설계

CI에서 생성된 .next는 불변 산출물이다.

-   동일 커밋 → 동일 빌드 결과
-   서버 상태와 무관
-   재현 가능성 보장

이는 "서버 상태 의존형 배포"에서\
"불변 산출물 기반 배포"로의 전환이다.

------------------------------------------------------------------------

## 환경 분리 전략

| 구분  |  CI 환경 |    서버 환경 |
|---- |---- |---- |
| 목적   |           빌드/검증  | 실행 |
| devDependencies  | 필요       | 불필요 |
| 타입 검사  |       수행      |  수행하지 않음 |
| 리소스     |       일시적   |   지속적 |

> Build Once, Deploy Many

------------------------------------------------------------------------

# 7. 트래픽 운영 관점에서의 실무 회고

## 서버 리소스 안정성

서버에서 빌드할 경우:

-   CPU 급상승
-   메모리 사용량 증가
-   트래픽 응답 지연 가능성

CI 빌드 구조로 변경 후:

-   서버는 실행만 담당
-   리소스 변동 최소화
-   예측 가능한 성능 유지

------------------------------------------------------------------------

## 장애 전파 차단

이전 구조:

코드 문제 → 서버 빌드 실패 → 서버 영향

현재 구조:

코드 문제 → CI 실패 → 배포 차단 → 서버 안전

------------------------------------------------------------------------

## 배포는 기능이 아니라 안전장치

CI/CD는 단순 자동화 도구가 아니라\
운영 리스크를 줄이는 안전장치다.

------------------------------------------------------------------------

# 마무리

이번 경험을 통해 배운 것은 단순 스크립트 작성법이 아니다.

-   빌드와 런타임 분리 전략
-   의존성 관리의 중요성
-   실패 감지 설계
-   불변 산출물 기반 배포
-   운영 안정성을 고려한 구조 설계

CI/CD는 배포 도구가 아니라\
아키텍처의 일부라는 것을 이해하게 되었다.
