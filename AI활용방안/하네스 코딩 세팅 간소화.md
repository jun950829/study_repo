너는 Claude Code 하네스 세팅 담당자야.

목표:
`/Users/Desktop/projects/example` 아래의 multi-repo 프로젝트를 Claude Code native Agent Teams 중심 하네스로 세팅해줘.

대상 repo:
- example-backend
- example-backend-admin
- example-backend-user
- example-frontend-admin
- example-frontend-user

핵심 방향:
- Claude Code native Agent Teams를 실제 협업 엔진으로 사용한다.
- tmux는 agent를 수동 연결하는 장치가 아니라 monitor/관제 화면으로만 사용한다.
- CEO는 IDE terminal에서 사용자가 주는 업무를 분석하고, 필요한 agent만 native Agent Teams로 소집한다.
- 기본은 pipeline 방식이고, 필요한 경우에만 concurrent meeting을 사용한다.
- 공유 상태는 `team-plan.md`, `decisions.md`, `handoff.md`, 회의록 중심으로 관리한다.
- 모든 agent가 output 파일을 만들 필요는 없다.
- `optional-outputs/`는 긴 API 분석, QA 결과, Security 리뷰, MR 초안처럼 보존 가치가 큰 산출물에만 사용한다.
- Codex는 구현 전 계획 리뷰와 구현 후 검증에 사용한다.
- Notion 연동은 아직 없다고 가정하고 Notion 기록 단계는 스킵한다.

Agent 역할:
- ceo: 전체 조율, 작업 분류, native Agent Teams 구성, Codex 게이트, 회의록 최종 책임
- pm-agent: 요구사항, 범위, 우선순위, 인수 조건
- architect: 시스템 구조, repo 경계, API 계약, 데이터 흐름, 배포 전략
- code-expert: 코드 품질, 패턴 일관성, 리팩터링 위험, diff 리뷰
- designer: UX/UI, Figma MCP 분석, 코드 UI 비교, 상태 설계
- notion-expert: Notion 연동 전까지 Markdown 회의록 정리와 향후 Notion 이관 가능한 구조화
- pr-expert: GitLab MR 초안, diff 요약, 리뷰 체크리스트, pipeline 상태 조회
- front-dev: admin/user frontend 구현
- back-dev: backend/admin/user backend 구현
- security: 인증, 권한, secret, 로그, API 보안 검토
- qa-engineer: 인수 조건, 테스트 계획, 회귀 검증
- customer-voice: 사용자 관점, CS 영향, 안내 문구, 릴리즈 커뮤니케이션

스크립트 요구사항:
1. `create-team-run.sh`
   - `.claude/runs/YYYY-MM-DD-task-slug/` 생성
   - `README.md`, `status.tsv`, `team-plan.md`, `decisions.md`, `handoff.md`, `codex/`, `optional-outputs/` 생성
   - `.claude/meetings/YYYY-MM-DD-task-slug.md` 생성
   - `.claude/tmux/current-run` 기록
   - `AGENTS`와 `MODE` 환경변수 지원

2. `start-harness-monitor.sh`
   - tmux session을 열고 현재 run, status.tsv, shared files, optional outputs, tmux panes, 최근 회의록, repo별 git status 표시
   - agent를 수동 실행하거나 prompt dispatch하지 않음

CLAUDE.md 규칙:
- 사용자가 작업을 주면 CEO가 직접 처리 / 단일 agent / native Agent Teams 중 선택한다.
- CEO는 필요한 경우 `create-team-run.sh`를 실행한다.
- CEO는 native Agent Teams로 필요한 agent만 teammate로 초대한다.
- 구현 전에는 Codex 계획 리뷰 전 단계에서 멈춘다.
- 구현 후에는 QA/Security와 Codex 검증을 거친다.
- git push, MR 생성, reviewer 지정, label 변경, merge는 사용자 승인 없이 하지 않는다.
- secret, token, .env, private key는 읽거나 출력하지 않는다.

HTML 문서:
- `EXAMPLE_HARNESS_PROCESS.html`을 생성한다.
- 전체 흐름, 아키텍처, shared files, 스크립트 역할, 실제 사용법, CEO 기본 프롬프트를 포함한다.

최종적으로 생성한 파일 목록과 사용법을 요약해줘.
