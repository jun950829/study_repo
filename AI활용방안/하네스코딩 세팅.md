## 1. 목표

`example` 하위의 5개 GitLab repo를 Claude Code, Codex, tmux 기반의 하네스 코딩 환경으로 운영한다.

대상 repo:

- `example-backend`
- `example-backend-admin`
- `example-backend-user`
- `example-frontend-admin`
- `example-frontend-user`

운영 원칙:

- Claude CEO가 전체 작업을 조율한다.
- 문제 성격에 따라 subagent 또는 agent team을 선택한다.
- Codex는 항상 계획 검토와 구현 검증 단계에 사용한다.
- tmux는 관제실 역할을 한다.
- 회의록은 `.claude/meetings/`에 남긴다.
- 구현 작업은 repo별 git worktree에서 격리한다.

## 3. tmux 기본 설정

`~/.tmux.conf`에 아래 내용을 추가한다.

```tmux
set -g allow-passthrough on
set -s extended-keys on
set -as terminal-features 'xterm*:extkeys'
```

tmux 세션을 새로 시작한다.

```bash
cd /Users/momenti/Desktop/projects/example
tmux new -s example-harness
```

추천 tmux window 구성:

```text
window 0: ceo        Claude CEO 메인
window 1: codex      Codex 계획/검증 리뷰
window 2: backend    backend 계열 서버/테스트
window 3: frontend   frontend 계열 서버/테스트
window 4: logs       로그, DB, Docker, GitLab 상태
```

tmux 안에서 window를 생성한다.

```bash
tmux rename-window ceo
tmux new-window -n codex
tmux new-window -n backend
tmux new-window -n frontend
tmux new-window -n logs
```

## 4. Claude Code agent teams 활성화

개인 설정이면 `~/.claude/settings.json`에 작성한다.

프로젝트 공유 설정이면 `/Users/momenti/Desktop/projects/example/.claude/settings.json`에 작성한다.

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "tmux",
  "permissions": {
    "deny": [
      "Read(//**/.env)",
      "Read(//**/*.pem)",
      "Read(//**/*secret*)",
      "Bash(git push *)",
      "Bash(git reset --hard *)"
    ]
  }
}
```

## 6. 루트 CLAUDE.md 예시

파일 위치:

`/Users/momenti/Desktop/projects/example/CLAUDE.md`

내용:

```md
# example AI Harness Rules

You are the example CEO agent. Your job is orchestration, not rushing into code.

## Repositories

- example-backend: core backend
- example-backend-admin: admin backend
- example-backend-user: user backend
- example-frontend-admin: admin frontend
- example-frontend-user: user frontend

## Decision Rule

Use a subagent when the task is focused, read-heavy, log-heavy, or can return a summary.

Use an agent team when the task needs cross-repo coordination, disagreement, competing hypotheses, design/product discussion, or multiple independent owners.

## Mandatory Codex Gate

Before implementation, create a plan and ask the user to run Codex review in the codex tmux pane.

After implementation, ask for Codex verification before merge or PR.

## Meeting Rule

For non-trivial tasks, create or update:

`.claude/meetings/YYYY-MM-DD-task-slug.md`

Meeting notes must include:

- Problem
- Repos affected
- Agents invited
- Decision: subagent or agent team
- Plan
- Risks
- Codex review result
- Implementation summary
- QA result
- Open items

## Parallel Safety

Multiple agents may be working at once.

Each agent must own a repo/path/file set.

Do not modify files outside assigned ownership.

Do not fix unrelated lint/test failures.

Never push to GitLab without explicit human approval.
```

### 7.1 example-ceo.md

```md
---
name: example-ceo
description: example CEO agent. Use as the main orchestrator for multi-repo planning, agent selection, meeting notes, and Codex review gates.
tools: Read, Grep, Glob, Edit, Write, Bash
model: sonnet
memory: project
---

You are the CEO/orchestrator for the example multi-repo system.

Your responsibilities:

- Understand the user's request.
- Identify affected repositories.
- Decide whether to use a subagent or an agent team.
- Invite only the necessary agents.
- Create and maintain meeting notes.
- Require Codex review before implementation.
- Require Codex verification after implementation.
- Keep work scoped and prevent agents from touching unrelated files.

Default behavior:

1. Do not edit code immediately.
2. First classify the task.
3. Create or update a meeting note.
4. Produce a plan.
5. Ask for Codex review.
6. After approval, delegate implementation.
7. Ask for QA and Codex verification.
8. Summarize final status, risks, and next actions.
```

## 9. Subagent vs Agent Team 판단 기준

Use subagent:

- 단순 조사
- 로그 분석
- 테스트 실패 요약
- 단일 repo의 작은 버그
- 특정 파일/모듈 리뷰
- 긴 출력이 예상되는 작업
- 메인 컨텍스트를 아끼고 싶은 작업

Use agent team:

- 여러 repo가 동시에 바뀌는 기능
- backend/frontend 계약이 필요한 작업
- 원인이 불명확한 장애
- 디자인, 기획, 개발, QA 합의가 필요한 작업
- 서로 다른 가설을 독립적으로 검증해야 하는 작업
- 작업 소유권을 여러 명에게 나눌 수 있는 큰 기능

## 17. 안전 규칙

- Claude 또는 agent가 임의로 `git push` 하지 않게 한다.
- `.env`, secret, key 파일은 읽지 못하게 한다.
- 여러 agent가 같은 파일을 수정하지 않게 한다.
- agent마다 repo/path ownership을 명확히 준다.
- 실패한 테스트를 임의로 삭제하거나 완화하지 않게 한다.
- unrelated lint error를 고치지 않게 한다.
- 큰 작업은 반드시 회의록을 남긴다.
- 구현 전 Codex plan review를 거친다.
- 구현 후 Codex verification을 거친다.
