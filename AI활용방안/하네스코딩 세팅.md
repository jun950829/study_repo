## 1. 목표

`example` 하위의 5개 GitLab repo를 Claude Code, Codex, tmux 기반의 하네스 코딩 환경으로 운영한다.

대상 repo:

- `example-backend`
- `example-backend-admin`
- `example-backend-user`
- `example-frontend-admin`
- `example-frontend-user`

운영 원칙:

- Claude CEO가 전체 작업을 조율한다.
- 문제 성격에 따라 단일 에이전트 또는 agent team을 선택한다.
- Codex는 항상 계획 검토와 구현 검증 단계에 사용한다.
- tmux는 관제실이자 하네스 대시보드 역할을 한다.
- 회의록은 `.claude/meetings/`에 남긴다.
- 구현 작업은 repo별 git worktree에서 격리한다.
- 보안, QA, PR, 문서화, 고객 관점을 역할별로 분리한다.

## 3. tmux 기본 설정

`~/.tmux.conf`에 아래 내용을 추가한다.

```tmux
set -g allow-passthrough on
set -s extended-keys on
set -as terminal-features 'xterm*:extkeys'
```

tmux 세션을 새로 시작한다.

```bash
cd /Users/Desktop/projects/example
tmux new -s example-harness
```

## 5. Claude Code agent teams 활성화

개인 설정이면 `~/.claude/settings.json`에 작성한다.

프로젝트 공유 설정이면 `/Users/Desktop/projects/example/.claude/settings.json`에 작성한다.

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "tmux",
  "permissions": {
    "deny": [
      "Read(//**/.env)",
      "Read(//**/.env.*)",
      "Read(//**/*.pem)",
      "Read(//**/*.key)",
      "Read(//**/*secret*)",
      "Read(//**/*credential*)",
      "Bash(git push *)",
      "Bash(git reset --hard *)",
      "Bash(git clean *)"
    ]
  }
}
```

## 7. 역할별 agent 파일

Agent 파일 위치:

```text
/Users/Desktop/projects/example/.claude/agents/
```

필요한 agent 파일:

```text
ceo.md

pr-expert.md
front-dev.md
back-dev.md

```

역할 요약:

| Agent | 역할 |
| --- | --- |
| `ceo` | 전체 조율, 회의 소집, 작업 소유권 배정, Codex 게이트 관리 |
| `pm-agent` | 요구사항, 범위, 우선순위, 인수 조건 정리 |
| `architect` | repo 경계, API 계약, 데이터 흐름, 배포 전략 설계 |
| `code-expert` | 코드 품질, 패턴 일관성, 리팩터링 위험, diff 리뷰 |
| `designer` | UX/UI, Figma MCP 분석, 코드 UI 비교, 상태 설계 |
| `notion-expert` | 회의록, 결정 기록, 제품/기술 문서화 |
| `pr-expert` | GitLab MR 설명, diff 요약, 리뷰 체크리스트 |
| `front-dev` | admin/user frontend 구현 |
| `back-dev` | backend/admin/user backend 구현 |
| `security` | 인증, 권한, secret, 로그, API 보안 검토 |
| `qa-engineer` | 인수 조건, 테스트 계획, 회귀 검증 |
| `customer-voice` | 사용자 관점, CS 영향, 안내 문구, 릴리즈 커뮤니케이션 |

## 8. 역할별 사용 기준

단일 에이전트 사용:

- 요구사항 정리만 필요하면 `pm-agent`
- 시스템 구조나 API 계약 검토가 필요하면 `architect`
- 코드 품질이나 diff 리뷰가 필요하면 `code-expert`
- Figma 또는 UI 분석이 필요하면 `designer`
- Notion 문서화가 필요하면 `notion-expert`
- MR 설명이 필요하면 `pr-expert`
- frontend 구현이 필요하면 `front-dev`
- backend 구현이 필요하면 `back-dev`
- 보안 검토가 필요하면 `security`
- QA 검증이 필요하면 `qa-engineer`
- 사용자/CS 관점이 필요하면 `customer-voice`

Agent team 사용:

- 여러 repo가 동시에 영향을 받는 기능
- backend와 frontend 간 API 계약 조율이 필요한 작업
- PM, 디자인, 개발, 보안, QA 관점이 함께 필요한 작업
- 원인이 불명확한 장애나 복잡한 버그
- 작업 소유권을 여러 에이전트에게 나눌 수 있는 큰 작업
- Figma 디자인, API, 데이터, 권한, 테스트 전략이 함께 얽힌 작업

## 10. meeting window 구성

`meeting` window는 필요할 때 agent team을 띄우는 회의실이다.

4분할 예시:

```bash
tmux select-window -t meeting
tmux split-window -h -c /Users/Desktop/projects/example
tmux split-window -v -t 0 -c /Users/Desktop/projects/example
tmux split-window -v -t 1 -c /Users/Desktop/projects/example
tmux select-layout tiled
```

각 pane에서 필요한 역할만 실행한다.

```bash
claude --agent pm-agent
claude --agent architect
claude --agent designer
claude --agent qa-engineer
```

회의실 사용 원칙:

- 모든 agent를 항상 띄우지 않는다.
- CEO가 필요한 역할만 선택한다.
- 각 agent에게 repo, 파일, 질문, 하지 말아야 할 작업을 명확히 준다.
- 회의 결과는 `.claude/meetings/YYYY-MM-DD-task-slug.md`에 기록한다.

## 13. status window 현황판

`status` window는 tmux 기반 하네스 대시보드다.

전체 pane 상태:

```bash
tmux list-panes -a -F '#S:#I.#P #{window_name} #{pane_current_path} #{pane_current_command}'
```

repo git 상태:

```bash
git -C example-backend status --short
git -C example-backend-admin status --short
git -C example-backend-user status --short
git -C example-frontend-admin status --short
git -C example-frontend-user status --short
```

회의록 목록:

```bash
ls -lt .claude/meetings
```

`watch`가 있다면:

```bash
watch -n 3 "tmux list-panes -a -F '#S:#I.#P #{window_name} #{pane_current_path} #{pane_current_command}'"
```

macOS에 `watch`가 없으면:

```bash
brew install watch
```

## 17. Repo별 worktree 운용

각 repo에서 큰 작업은 worktree로 분리한다.

예시:

```bash

claude --worktree feature-auth-backend --tmux
```

```bash
cd /Users/Desktop/projects/example/example-frontend-user

```

```bash

```

## 18. 전체 작업 루틴

1. tmux 관제실을 시작한다.

```bash
cd /Users/Desktop/projects/example
tmux new -s example-harness
```

2. `ceo` window에서 CEO를 실행한다.

```bash
claude \
  --agent ceo \
  --add-dir ./example-backend \
  --add-dir ./example-backend-admin \
  --add-dir ./example-backend-user \
  --add-dir ./example-frontend-admin \
  --add-dir ./example-frontend-user
```

3. CEO에게 작업을 요청한다.

```text
example CEO로 동작해줘.
이 작업을 직접 처리할지, 단일 에이전트가 적합한지, agent team 회의가 필요한지 판단해줘.
영향 받는 repo, 필요한 역할, 작업 소유권, Codex 리뷰용 계획을 먼저 작성해줘.
아직 코드는 수정하지 마.
```

4. `codex` window에서 계획 리뷰를 수행한다.

5. CEO에게 Codex 리뷰 결과를 전달한다.

6. 필요한 역할을 `meeting`, `backend`, `frontend` window에서 실행한다.

7. 구현 후 `qa-engineer`가 검증한다.

8. 보안 영향이 있으면 `security`가 검토한다.

9. `codex` window에서 최종 구현 검증을 수행한다.

10. MR이 필요하면 `pr-expert`가 MR 설명을 작성한다.

11. 문서화가 필요하면 `notion-expert`가 회의 결과와 결정 사항을 정리한다.

12. 사용자 영향이 있으면 `customer-voice`가 안내 문구와 CS 영향을 검토한다.

13. 사람이 diff 확인 후 GitLab MR 또는 push를 승인한다.

## 19. 안전 규칙

- Claude 또는 agent가 임의로 `git push` 하지 않게 한다.
- `.env`, secret, key 파일은 읽지 못하게 한다.
- 여러 agent가 같은 파일을 동시에 수정하지 않게 한다.
- agent마다 repo/path ownership을 명확히 준다.
- 실패한 테스트를 임의로 삭제하거나 완화하지 않게 한다.
- 관련 없는 lint error를 고치지 않게 한다.
- 큰 작업은 반드시 회의록을 남긴다.
- 구현 전 Codex plan review를 거친다.
- 구현 후 Codex verification을 거친다.
- 보안 영향이 있는 작업은 `security` 검토를 거친다.
